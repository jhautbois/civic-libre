# État de l'art : bases libres pour une application citoyenne municipale

Date de l'étude : 7 juillet 2026. Ce document est le livrable de la phase 0 du projet (voir docs/spec.md).

Méthode : 9 recherches web parallèles (une par candidat ou domaine), menées en français et en anglais sur les sources primaires (dépôts Git, RFC, CNIL, sites officiels), suivies d'une vérification adverse : pour chaque candidat, un agent indépendant a tenté de réfuter les affirmations décisives en retournant aux sources. Les verdicts de vérification sont indiqués ci-dessous. Les limites et incertitudes sont signalées plutôt que masquées.


## Communecter (Open Atlas / Pixel Humain)

**Verdict : Écarter. Le projet est sincèrement vivant (commits de juillet 2026 sur GitLab Adullact) et l'ancrage français est réel, mais c'est une fausse bonne piste pour ce cahier des charges : la couverture des quatre besoins est faible (agenda sans iCal, aucun signalement Open311, pas de Web Push, back-office générique sans routage de tickets), la stack Yii 1/PHP 7/MongoDB 3.6 est en fin de vie, la licence est Apache-2.0 et non AGPL, le mobile est abandonné, et le déploiement multi-dépôts est trop complexe pour une maintenance par des tiers dans une commune de 7000 habitants sans DSI. Communecter est un réseau social territorial et une plateforme d'appels à projets, pas une app citoyenne municipale : l'adopter imposerait de développer l'essentiel des besoins au-dessus d'une base legacy portée par une très petite équipe.**

- Licence : Apache-2.0. Confirmée par le fichier LICENSE (texte intégral Apache License Version 2.0) sur le dépôt actif gitlab.adullact.net/pixelhumain/co2 (branche master) et par le label du miroir GitHub pixelhumain/co2. Attention : le wiki ADEME Résilience des Territoires mentionne « AGPL », mais le dépôt fait foi. Ce n'est donc PAS un copyleft fort, contrairement à la préférence AGPL-3.0 du projet.
- Activité : Projet vivant en 2026, mais attention au piège : les dépôts GitHub (pixelhumain/co2, pixelhumain/communecter) sont gelés depuis 2017-2018 (dernière release « Yellow Submarine » 26/11/2018). Le développement réel est sur gitlab.adullact.net/pixelhumain : dernier commit co2 le 6 juillet 2026 (« feat: Export coform to PDF »), ~12 000 commits, 27 sous-projets actifs en 2025-2026. Maintenu par Open Atlas, SCIC (coopérative) basée à La Réunion depuis 2021. Communauté très restreinte : 21 stars GitHub, 5 stars GitLab, 2-3 committers visibles récents (cael, mirana), bus factor faible.
- Pile technique : Cœur : PHP 7 sur framework Yii 1.x (mongoYii) + MongoDB 3.6/4.2 + jQuery/Bootstrap. Architecture éclatée en ~15 dépôts couplés (pixelhumain, co2, citizenToolKit, costum, map, survey, news, interop, api...). Composants récents en marge : notif-hub en Node.js 20/TypeScript (Fastify, Redis, PostgreSQL, BullMQ) pour le push mobile. La stack cœur est en fin de vie : PHP 7 EOL nov. 2022, MongoDB 3.6 EOL avril 2021, Yii 1 abandonné en amont.
- Hébergement : Docker officiel existant (gitlab.adullact.net/pixelhumain/docker, dernière activité février 2025) mais orienté développement : le README s'intitule « installer php7 et mongo 3.6 pour dev sur ubuntu ». docker-compose à 3 services (mongo:3.6, mongo:4.2, front Apache/PHP7), puis clonage manuel d'environ 15 dépôts dans code/, composer install dans le conteneur, écriture manuelle de dbconfig.php, chmod 777 sur les assets. Pas d'image de production clé en main, pas de branche PHP 8 (la branche php7 a été fusionnée dans master). Complexité élevée et fragile pour une commune sans service informatique.
- Couverture du besoin : agenda : Partiel. Agenda d'événements géolocalisés natif (AgendaController, EventController). RSS réel : citizenToolKit/components/RssGenerator.php gère events, news, organizations, projects, poi, classifieds. iCal ABSENT : controllers/convert/ICalendarAction.php est un fichier vide de 0 octet (blob e69de29, placeholder jamais implémenté). Import OpenAgenda présent (sens import uniquement). ExportController ne propose que CSV et PDF.
  signalement : Absent. Aucun module de signalement citoyen, aucune trace d'Open311 dans les arborescences complètes de co2, citizenToolKit et costum (scan des fichiers via l'API GitLab). Le module « signalement » le plus proche est le kanban du module Appels à Projets, sans rapport avec le cas d'usage voirie/photo/géoloc/catégorie.
  notifications : Partiel et inadapté. Notifications in-app (NotificationController : getnotifications, marknotificationasread) et e-mail. Le nouveau notif-hub (Node/TS, actif 2026) est un serveur de push FCM/APNs pour apps natives : pas de Web Push VAPID trouvé nulle part dans l'écosystème. Pas de service worker ni manifest.json dans co2 : pas de canal PWA.
  back-office : Partiel. Administration générique (AdminController) avec gestion fine des rôles et communautés, kanban pour les appels à projets. Pas de file de tickets avec routage vers services municipaux ni statuts à traiter/en cours/résolu au sens Open311.
- Standards (iCal, RSS, Open311, Web Push) : RSS : oui (RssGenerator.php, code réel vérifié). iCal : non (fichier d'action vide, jamais implémenté). Open311 : non (aucune occurrence dans le code). Web Push VAPID : non (notif-hub ne fait que FCM/APNs natif ; pas de service worker dans co2). Interop présente mais ailleurs : ActivityPub (ActivitypubController), Discourse, GitLab, MediaWiki, OSM, Wikidata, import OpenAgenda.
- Français : Excellent sur ce critère : projet français (Open Atlas, SCIC à Saint-Louis, La Réunion), interface nativement en français, documentation en français (doc.co.tools, toutefois inaccessible lors du test : connexion refusée), dépôts hébergés sur la forge publique française GitLab Adullact, instance principale communecter.org.
- Ergonomie : Point faible. Application mobile Cordova abandonnée : v0.0.18 du 4 septembre 2018 (APKPure), retirée du Play Store (fiche en 404). Des chantiers mobiles récents existent (cocolight actif avril 2026, coeurmobi-monorepo) mais rien d'abouti publiquement. Pas de PWA au sens strict (aucun manifest.json ni service worker trouvés dans co2). UI web jQuery/Bootstrap dense, pensée réseau social cartographique de bureau, pas mobile-first. Aucune mention de conformité RGAA trouvée.
- Risques :
  - Stack cœur en fin de vie : Yii 1 + PHP 7 + MongoDB 3.6/4.2, dette de sécurité et de maintenance majeure pour un déploiement public RGPD
  - Licence Apache-2.0 permissive, en contradiction avec l'exigence de copyleft fort AGPL-3.0
  - Bus factor très faible : petite équipe d'Open Atlas (2-3 committers récents), communauté quasi inexistante hors de la coopérative (5 stars GitLab, 21 GitHub)
  - Architecture éclatée en ~15 dépôts couplés, installation dev-oriented fragile (chmod 777, configs manuelles), documentation technique doc.co.tools inaccessible lors du test
  - Inadéquation fonctionnelle profonde : réseau social territorial généraliste + appels à projets, sans signalement Open311, sans iCal, sans Web Push, mobile abandonné : il faudrait développer l'essentiel des besoins dans une base legacy
  - Aucune preuve d'utilisation actuelle par une petite commune métropolitaine comme application citoyenne : la référence active est Saint-Denis de La Réunion (150 000 hab.) pour les appels à projets, un autre métier
  - Aucune mention de conformité RGAA ni de démarche accessibilité trouvée

## DirectMairie (ADULLACT)

**Verdict : Brique à assembler, avec réserves fortes (à la limite de l'écarter). Points pour : licence AGPL-3.0 exactement conforme à la contrainte, code propre et techniquement à jour (Angular 21, Spring Boot 4, PostgreSQL, renovate), périmètre signalement bien construit (photo, géoloc, catégories, anonymat, multi-collectivités, back-office simple avec résolution/rejet motivés). Points contre décisifs : il ne couvre qu'un des besoins (signalement) et sans le standard Open311 exigé, sans Web Push, sans agenda ni iCal/RSS ; aucune image Docker officielle ; et surtout le service mutualisé ADULLACT paraît arrêté sans annonce, avec un développement fonctionnel gelé depuis septembre 2023 et une communauté quasi nulle. En faire la base unique de l'application citoyenne demanderait d'écrire l'agenda, l'API Open311 et le push, c'est-à-dire l'essentiel du projet. Dans l'écosystème ADULLACT/Comptoir du Libre, les autres briques pertinentes sont : Publik (Entr'ouvert, AGPL-3.0, 21 collectivités déclarées, GRC/portail citoyen complet mais plateforme multi-services lourde à opérer pour une commune de 7000 habitants sans DSI), Lutèce + plugin DansMaRue (BSD, éprouvé à Paris mais copyleft faible et dimensionné grandes villes), GeoContrib (AGPL, signalement géographique collaboratif mais UX non grand public et sans Open311), Décidim/CONSUL (participation citoyenne, hors sujet agenda/signalement opérationnel) ; aucun agenda communal libre n'est référencé au Comptoir. Recommandation : ne retenir DirectMairie que comme brique signalement à forker (ou comme référence de modèle de données) après avoir clarifié son statut auprès de l'ADULLACT ; si Open311 est une exigence dure, une base type FixMyStreet Platform (hors écosystème ADULLACT) mérite la comparaison avant décision.**

- Licence : GNU AGPL-3.0 (fichier LICENSE à la racine du dépôt, vérifié en raw : https://gitlab.adullact.net/directmairie/directmairie/-/raw/master/LICENSE ; confirmé par la fiche Comptoir du Libre « GNU aGPL v3 »). Conforme à la préférence copyleft fort du projet.
- Activité : Dernier commit : 3 décembre 2025 (upgrade Spring Boot 4 par jnizet, puis renovate[bot]), vérifié via l'API GitLab. Mais 27 des 30 derniers commits sont des mises à jour automatiques de dépendances (renovate[bot]) mergées par jnizet et cexbrayat (Ninja Squad, prestataire de l'ADULLACT). Dernière release fonctionnelle : 3.0.2 du 20 septembre 2023 ; aucune évolution fonctionnelle depuis. 1760 commits, 5 releases, dépôt créé en décembre 2018. Communauté quasi inexistante : 2 utilisateurs déclarés au Comptoir du Libre (aucune commune nommée). Signal d'alerte majeur : au 7 juillet 2026, l'instance mutualisée directmairie.adullact.org refuse les connexions (testé par curl local et WebFetch), la page produit adullact.org/web-service-directmairie renvoie HTTP 500, et DirectMairie n'apparaît plus dans la liste des services sur adullact.org ; aucune annonce officielle d'arrêt n'a été trouvée (la FAQ existe toujours).
- Pile technique : Backend : Kotlin sur Spring Boot 4 (JVM, Java 17+), PostgreSQL, migrations Flyway, JWT (jjwt), envoi d'emails via spring-boot-starter-mail avec templates Mustache, géocodage inverse Nominatim. Frontend : Angular 21, Bootstrap 5, Leaflet (ngx-leaflet) pour la carte, @angular/service-worker (PWA), pnpm, tests Playwright/Karma. Build : Gradle multi-modules (Kotlin DSL). CI GitLab + CircleCI, renovate actif.
- Hébergement : Pas d'image Docker officielle ni de docker compose de production : le docker-compose.yml du dépôt est un environnement de développement (postgres:18-alpine en mode trust + Mailhog). Déploiement officiel documenté : jar Spring Boot + PostgreSQL + serveur SMTP (Documentation/For_operators, qui mentionne encore OpenJDK 11 alors que le code exige désormais Java 17+, doc obsolète), plus un module Puppet officiel (Puppet-DirectMairie, dernière activité juin 2025). Un packaging docker compose de production serait à écrire soi-même (faisable : app JVM monolithique + PostgreSQL + SMTP). Auto-hébergement FR/UE possible sans dépendance externe autre que Nominatim (OSM).
- Couverture du besoin : agenda : Absent. Aucune fonctionnalité agenda ou événements, pas d'iCal ni de RSS nulle part dans le dépôt. DirectMairie ne couvre pas du tout ce besoin, et le Comptoir du Libre ne référence aucun outil d'agenda communal libre (ni Mobilizon ni Gancio n'y figurent).
  signalement : Cœur du produit : signalement géolocalisé (position GPS ou clic sur carte Leaflet), catégories et sous-catégories configurables par collectivité (IssueCategory/IssueGroup), description, photo optionnelle, dépôt anonyme ou avec compte, suivi par le citoyen, export CSV. Multi-collectivités avec filtrage EPCI. PAS de standard Open311 (vérifié : aucune occurrence dans l'arborescence backend, les contrôleurs REST ni les dépendances Gradle).
  notifications : Email uniquement : notification de la collectivité à la création d'un signalement (new-issue.mustache), notification du citoyen lors des transitions résolu/rejeté avec motif (issue-transition.mustache), et alertes email de la mairie vers les citoyens abonnés (menu alertes, ajouté en 3.0.0 d'avril 2023). Aucun Web Push : pas de bibliothèque VAPID côté backend, le service worker Angular ne sert que le cache PWA.
  back-office : Back-office web pour les admins de collectivité : traitement des signalements (résoudre ou rejeter avec motif associé), configuration des catégories/sous-catégories, export CSV, envoi d'alertes, rôles admin/superadmin, gestion multi-communes et EPCI. Limites : statuts en base limités à DRAFT/CREATED/RESOLVED/REJECTED (vérifié dans domain/Issue.kt), donc pas de statut « en cours », et pas de routage/affectation vers un service interne (voirie, espaces verts...) ; le routage se limite à la notification email de la collectivité concernée.
- Standards (iCal, RSS, Open311, Web Push) : iCal : non. RSS : non. Open311 (GeoReport v2) : non, aucune trace dans le code, la doc ou les dépendances (vérifié via l'arborescence complète backend/src/main et build.gradle.kts ; la recherche code de l'API GitLab exige une authentification, mais les noms de fichiers et dépendances sont sans ambiguïté). Web Push/VAPID : non, notifications exclusivement par email. Le seul standard réellement présent est une API REST propre documentée par Spring REST Docs, plus l'usage de Nominatim/OSM.
- Français : Produit conçu par et pour les collectivités françaises (ADULLACT, Montpellier) : interface citoyenne et back-office en français, FAQ et documentation utilisateur en français (faq.adullact.org). La documentation développeur/exploitant du dépôt est en anglais (dossiers For_developers/For_operators). Hébergement historique de l'instance mutualisée en France par l'ADULLACT, mais cette instance est injoignable au 7 juillet 2026. Applications mobiles publiées sur App Store et Play Store fin 2022 (disponibilité actuelle non vérifiée ; le dépôt « DirectMairie Mobile » est sans activité depuis sa création en mai 2023).
- Ergonomie : Frontend Angular 21 + Bootstrap 5, pensé mobile-first pour le parcours citoyen (carte Leaflet, photo, géolocalisation), PWA via @angular/service-worker (cache hors-ligne). Design fonctionnel mais standard Bootstrap, pas de système de design travaillé. Accessibilité RGAA : aucune déclaration ni audit trouvés dans le dépôt ou la doc (point non vérifié, à considérer comme non couvert). Impossible de tester l'ergonomie réelle : l'instance de démonstration/production est hors ligne.
- Risques :
  - Le service mutualisé semble abandonné : instance directmairie.adullact.org injoignable (connexion refusée, vérifié le 2026-07-07 depuis deux réseaux), page produit adullact.org en erreur 500, DirectMairie retiré de la liste des services ADULLACT, sans annonce officielle. Le devenir du projet est incertain et il faut contacter l'ADULLACT avant tout engagement.
  - Aucun développement fonctionnel depuis la release 3.0.2 (20 septembre 2023) : l'activité récente est à 90 % du renovate[bot] ; le groupe de travail collaboratif semble dormant depuis 2023.
  - Couverture fonctionnelle très partielle par rapport au besoin : pas d'agenda, pas d'iCal/RSS, pas d'Open311, pas de Web Push, pas de statut « en cours », pas de routage par service. Tout cela serait du développement custom en Kotlin/Spring + Angular.
  - Pas d'image Docker officielle ni de compose de production : packaging à créer, doc d'exploitation obsolète (OpenJDK 11 annoncé alors que le code exige Java 17+), déploiement officiel pensé pour Puppet.
  - Adoption très faible et non documentée : 2 utilisateurs déclarés au Comptoir du Libre, aucune commune identifiable publiquement ; en cas de reprise, la commune serait quasi seule sur le code.
  - Accessibilité RGAA et conformité RGPD non documentées dans le dépôt (pied de page RGPD ajouté en 2.2.0, mais aucun registre ni déclaration d'accessibilité trouvés).

## Decidim (decidim.org) et CONSUL Democracy (consuldemocracy.org), évalués conjointement en tant que plateformes de participation citoyenne

**Verdict : Écarter pour ce périmètre, les deux. Ce sont d'excellentes plateformes de participation démocratique (leur vrai métier : budgets participatifs, consultations, assemblées), pas des applications agenda + signalement. Sur les deux besoins cœur : agenda communal grand public avec iCal/RSS/push, couverture partielle chez Decidim seulement (Meetings + push VAPID, mais pensé réunions de concertation, pas vie locale, pas de RSS) ; signalement Open311 avec routage par service, couverture nulle chez les deux (pas d'Open311, back-office de modération et non de tickets). Le poids opérationnel (Rails + PostgreSQL + Redis + Sidekiq, upgrades à breaking changes, admin complexe) est celui d'une métropole outillée, pas d'une commune de 7000 habitants sans informaticien. Nuance : Decidim reste une brique optionnelle pour plus tard, en SaaS chez un prestataire, si la commune veut un jour lancer un budget participatif ; ne jamais l'auto-héberger pour ce seul besoin. CONSUL est à écarter sans réserve pour la France (pas d'agenda, pas de push, écosystème français inexistant).**

- Licence : AGPL-3.0 pour les deux. Decidim : AGPL-3.0 confirmée sur https://github.com/decidim/decidim. CONSUL Democracy : Affero GPL v3 confirmée sur https://github.com/consuldemocracy/consuldemocracy. Conforme à la préférence copyleft fort du projet.
- Activité : Les deux sont très actifs en 2026. Decidim : release v0.32.0 le 6 juillet 2026, ~8000 commits, 1,8k étoiles, maintenu par l'Association Decidim (Barcelone) avec une large communauté. CONSUL Democracy : release 2.5.1 le 4 juin 2026, ~21 000 commits, 1,5k étoiles, maintenu par la CONSUL Democracy Foundation, utilisé par plus de 100 institutions dans 35 pays. Aucun risque d'abandon côté code amont.
- Pile technique : Identique dans l'esprit : monolithes Ruby on Rails + PostgreSQL. Decidim : Ruby on Rails (multi-engines), PostgreSQL, Redis + Sidekiq pour les jobs, Node.js/Webpack pour les assets, SMTP requis. CONSUL : Ruby 3.4.9, Node.js 22, PostgreSQL >= 13, ImageMagick, CMake. Stack lourde et exigeante en compétences Ruby/Rails, très éloignée d'un simple conteneur à lancer.
- Hébergement : Point faible pour une commune sans informaticien. Decidim : le docker-compose fourni est présenté comme un exemple, la doc officielle précise qu'avec cette image on ne peut pas utiliser de modules tiers, et oriente la production vers une installation manuelle nginx/Passenger/rbenv réservée à qui « connaît Ruby on Rails » (docs.decidim.org). CONSUL : Dockerfile et docker-compose.yml présents dans le dépôt, mais la voie de production documentée est un installeur Ansible sur serveur nu Ubuntu/Debian (github.com/consuldemocracy/installer), sans Docker. Aucun des deux ne documente officiellement la RAM requise ; au vu de la stack (Puma + Sidekiq + PostgreSQL + Redis), il faut compter réalistement 2 à 4 Go, le guide Platoniq recommande d'ajouter 3 Go de swap rien que pour les déploiements. Coût VPS estimé 10 à 25 EUR/mois, mais le vrai coût est humain : les mises à jour Decidim se font version par version avec breaking changes (un cas documenté de migration 0.27 vers 0.28 a pris 6 mois, pearlthoughts.com). Sans prestataire, c'est intenable pour la cible.
- Couverture du besoin : agenda : Partiel chez Decidim uniquement : le composant Meetings offre vue calendrier, export iCalendar (RFC 5545) et lien Google Calendar (decidim.org/features), mais il est conçu pour des réunions rattachées à des espaces de participation (concertations, assemblées), pas comme agenda communal généraliste (fêtes, travaux, vie associative). Pas de flux RSS documenté. CONSUL : aucun module agenda/événements.
  signalement : Aucun support Open311 natif ni chez Decidim ni chez CONSUL (aucune mention dans les dépôts ni les docs). Decidim dispose d'un module tiers openpoke/decidim-module-reporting-proposals (formulaire mobile en une étape, photo via caméra, GPS en un clic, catégories, assignation d'évaluateurs) mais il ne parle pas Open311 et sa dernière compatibilité annoncée est Decidim 0.29.x alors que le core est en 0.32.0, soit 3 versions de retard. CONSUL : propositions géolocalisées sur carte, mais orientées idées citoyennes, pas signalement de voirie.
  notifications : Decidim : oui, Web Push avec clés VAPID depuis la v0.27 (PR #8738 et #8774, générateur de clés inclus), plus notifications internes et e-mail. C'est le point fort de Decidim sur ce besoin. CONSUL : notifications internes et e-mail seulement, pas de Web Push documenté.
  back-office : Les deux ont un back-office complet mais orienté modération de participation (utilisateurs, propositions, votes), pas gestion de tickets. Decidim propose l'assignation de valuators par catégorie et des états de réponse (en évaluation, acceptée, rejetée) qui simulent approximativement à traiter/en cours/résolu, mais pas de routage vers des services techniques municipaux. L'admin Decidim est réputée riche et complexe, disproportionnée pour des agents non formés d'une commune de 7000 habitants.
- Standards (iCal, RSS, Open311, Web Push) : iCal : oui chez Decidim (export des meetings, RFC 5545), non chez CONSUL. RSS : non documenté chez Decidim, non trouvé chez CONSUL (à considérer comme absent). Open311 : absent des deux, aucune trace dans les dépôts, docs ou modules officiels. Web Push VAPID : oui chez Decidim depuis la 0.27 (générateur de clés VAPID intégré), absent chez CONSUL. Bilan : Decidim couvre 2 standards sur 4, CONSUL 0 sur 4.
- Français : Decidim : interface intégralement traduite en français, très forte présence française historique (plus de 200 institutions accompagnées par Open Source Politics, de la petite commune à l'Assemblée nationale). MAIS événement majeur de 2026 : OSP, principal prestataire Decidim français, a été racheté le 31 mars 2026 par Go Vocal, éditeur propriétaire, suite à des difficultés financières ; l'Association Decidim a publiquement dénoncé ce basculement (decidim.org, blog du 7 avril 2026). Une partie de l'équipe rejoint LaSuite.coop (SCIC avec IndieHosters au 1er juillet 2026). L'écosystème français Decidim est donc en pleine recomposition. CONSUL : interface disponible en 40+ langues dont le français, mais docs essentiellement en anglais/espagnol et déploiements français quasi inexistants dans les sources consultées (implantation surtout Espagne et Amérique latine ; la mention de Paris n'a pas pu être vérifiée pour un usage actuel).
- Ergonomie : Decidim : bon niveau, redesign complet en 0.28, fonctionnalités PWA (Add to Home Screen, push), mobile correct, conformité accessibilité revendiquée WCAG 2.1 (pas de certification RGAA spécifique trouvée). CONSUL : interface fonctionnelle mais plus datée, pas de PWA ni push, accessibilité revendiquée mais non vérifiée RGAA. Dans les deux cas, l'expérience est pensée pour la participation (s'inscrire, proposer, voter), pas pour consulter rapidement l'agenda du village ou signaler un lampadaire en 30 secondes.
- Risques :
  - Inadéquation fonctionnelle de fond : les deux plateformes sont centrées budgets participatifs, consultations, propositions et votes ; l'agenda communal simple et le signalement Open311 n'y existent pas ou seulement par approximation (Meetings, module tiers).
  - Charge de maintenance disproportionnée : mises à jour Decidim version par version avec breaking changes (migration 0.27 vers 0.28 documentée à 6 mois d'effort), stack Rails complète à opérer ; irréaliste sans prestataire pour une commune sans service informatique.
  - Fragilité de l'écosystème français Decidim : rachat d'Open Source Politics par Go Vocal (propriétaire) en mars 2026, avertissement public de l'Association Decidim ; le support francophone se recompose (LaSuite.coop/IndieHosters) sans visibilité sur les tarifs petites communes.
  - Le seul module de signalement crédible (openpoke/decidim-module-reporting-proposals) est en retard de 3 versions sur le core (0.29 vs 0.32) et n'implémente pas Open311 : dépendance à un tiers peu garanti.
  - CONSUL : quasi-absence de communauté et de prestataires en France, pas de Web Push, pas d'agenda ; l'adopter reviendrait à porter seul un gros Rails espagnol pour n'utiliser aucune de ses fonctions clefs.

## Gancio (recommandé pour la brique agenda, comparé à Mobilizon, OpenAgenda, Agenda du Libre et Gathio)

**Verdict : Brique à assembler, à adopter telle quelle pour la fonction agenda : Gancio est la réponse à la question décisive, il expose nativement iCal (/feed/ics) et RSS (/feed/rss) vérifiés en production, avec zéro code de colle pour ces flux, une saisie anonyme modérée idéale pour les associations, la récurrence, le français complet, un docker officiel et une base SQLite sans administration : c'est la maintenance la plus faible du panel. Mobilizon : à écarter pour ce cas (trop lourd à opérer, modèle groupes/fédération surdimensionné), sauf si le Web Push natif VAPID devient le critère dominant. OpenAgenda : à écarter comme base (non libre, non auto-hébergeable, dépendance SaaS), éventuellement utilisable en canal de rediffusion open data. Agenda du Libre (AGPL, actif) : orienté communauté du libre, pas conçu comme agenda municipal générique. Gathio (GPL-3.0) : événements éphémères sans agenda permanent, hors besoin.**

- Licence : AGPL-3.0 (GNU AGPLv3 affichée sur le dépôt officiel https://framagit.org/les/gancio et sur https://gancio.org). Comparaison : Mobilizon est aussi AGPL-3.0 ; OpenAgenda n'est PAS libre (plateforme SaaS propriétaire, seuls les SDK/plugins WordPress GPL-2.0, Typo3, etc. sont publiés sur https://github.com/OpenAgenda).
- Activité : Très actif : dernier commit le 06/07/2026 (release 2.0.0-beta.5), release stable v1.28.2 le 14/12/2025, v1.28.0 le 08/09/2025 (vérifié via l'API Framagit, projet 48668, 4521 commits, 58 releases). Maintenu par underscore hacklab (dev principal « lesion »), communauté fediverse italienne/européenne. Mobilizon est également actif : repris fin 2024 par l'association française Kaihuri (avec la bénédiction de Framasoft, financement NLnet), v5.2.4 publiée le 30/06/2026, commits fin juin 2026 sur https://framagit.org/kaihuri/mobilizon. Agenda du Libre (agenda-libre-ruby) : actif, dernier commit 04/07/2026.
- Pile technique : Node.js : Nuxt/Vue + Vuetify côté front, Sequelize côté serveur, base SQLite par défaut (PostgreSQL/MariaDB possibles en v1.x). La v2 (beta, Nuxt 3/Nitro, TypeScript) est en cours. Empreinte très légère, un seul service. Comparaison : Mobilizon = Elixir/Phoenix + PostgreSQL avec PostGIS + front Vue, nettement plus lourd à opérer pour une commune sans service informatique.
- Hébergement : Auto-hébergeable simplement : image Docker officielle maintenue par l'auteur (cisti/gancio sur Docker Hub, référencée dans les release notes officielles et la doc https://gancio.org/install/docker), volume de données + fichier .env, nginx en reverse proxy, SQLite par défaut donc aucune base à administrer. Point d'attention : la doc docker actuellement en ligne (branche beta 2.0) montre un docker run simple plutôt qu'un docker-compose.yml complet ; la mise en compose reste triviale (un seul conteneur). Mobilizon : doc docker officielle avec docker compose (image kaihuri/mobilizon + conteneur PostgreSQL/PostGIS), plus de pièces mobiles. OpenAgenda : non auto-hébergeable, SaaS uniquement (hébergement UE géré par l'éditeur).
- Couverture du besoin : agenda : Excellente : agenda public partagé pensé exactement pour ce cas (événements locaux d'une ville), saisie anonyme activable avec file de modération, événements récurrents complets (hebdo, bimensuel, mensuel, ordinal, annuel : vérifié dans i18n/locales/fr.json, options admin allow_recurrent_event), tags, lieux, collections, multi-dates, import/embed par webcomponent ou iframe, plugin WordPress officiel dans le dépôt (wp-plugin).
  signalement : Absente : Gancio ne fait pas de signalement citoyen ni d'Open311. C'est une brique agenda pure, à assembler avec une brique signalement distincte.
  notifications : Partielle : RSS et iCal natifs pour l'abonnement, fédération ActivityPub (suivi depuis Mastodon etc.), emails de modération/confirmation, plugins (webhook, telegram). PAS de Web Push VAPID natif (aucune trace dans le code, la doc ou l'issue tracker). Mobilizon, lui, a le Web Push VAPID natif (doc officielle docs.mobilizon.org, section configure/push) mais ciblé utilisateurs connectés.
  back-office : Correcte pour l'agenda : rôles admin/éditeur, file de modération des événements anonymes, édition/suppression, options d'instance. Pas de notion de routage vers services municipaux ni de statuts de traitement (ce besoin relève de la brique signalement, hors périmètre Gancio).

ActivityPub, microdata/microformats en plus. Open311 : non (hors périmètre agenda). Web Push : non natif. Projet d'origine italienne, instances francophones existantes (liste sur gancio.org) mais je n'ai pas vérifié de déploiement par une collectivité française.   - Aucun audit d'accessibilité RGAA trouvé pour Gancio ni Mobilizon : conformité à vérifier et probablement à compléter.
  - Gancio ne couvre que l'agenda : le signalement Open311 et le routage back-office devront venir d'une autre brique, avec l'intégration correspondante.
  - OpenAgenda comme solution de repli : dépendance forte à un SaaS propriétaire, gratuit en entrée de gamme mais 3500 à 12000 euros HT/an pour les offres complètes, et pas d'auto-hébergement possible.

## Montage « Web Push natif (VAPID) dans la PWA municipale » + listmonk (courriel en masse) + RSS/iCal + API Mastodon ; ntfy réservé aux notifications internes des agents. Évaluation des briques libres de notification (Web Push, ntfy, Gotify, UnifiedPush, listmonk).

**Verdict : Brique à assembler. Le montage minimal et fiable pour une mairie de 7000 habitants, sans app store et sans compte chez un service tiers : (1) PWA communale auto-hébergée (docker compose, hébergeur FR/UE) intégrant Web Push VAPID directement dans son backend via web-push-php ou pywebpush, clés auto-générées, zéro service externe à contractualiser ; (2) listmonk (AGPL-3.0, très actif, UI en français) pour le canal courriel des habitants sans smartphone ou refusant le push ; (3) flux RSS et iCal générés par l'application, republiés vers Mastodon par un simple appel API en cron ; (4) éventuellement ntfy en interne pour alerter les agents. Gotify et UnifiedPush sont à écarter pour le grand public. Seule concession inévitable : le transport push transite chiffré par les infrastructures des éditeurs de navigateurs, ce qu'aucune solution standard ne permet d'éviter.**

- Licence : Briques à licences hétérogènes, toutes libres et compatibles avec un cœur AGPL-3.0 : web-push-php MIT (github.com/web-push-libs/web-push-php), pywebpush MPL-2.0 (github.com/web-push-libs/pywebpush), web-push Node MPL-2.0 (registry.npmjs.org/web-push v3.6.7), webpush-go MIT (github.com/SherClockHolmes/webpush-go), listmonk AGPL-3.0 (github.com/knadh/listmonk), ntfy Apache-2.0 (github.com/binwiederhier/ntfy), Gotify MIT (fichier LICENSE du dépôt gotify/server, GitHub affiche NOASSERTION), Apprise BSD-2-Clause (github.com/caronc/apprise). Aucune brique de notification n'est elle-même en copyleft fort sauf listmonk (AGPL-3.0).
- Activité : Vérifié via l'API GitHub le 7 juillet 2026 : web-push-php poussé le 2026-07-01 (très actif) ; pywebpush poussé le 2026-03-03 ; webpush-go poussé le 2026-04-22 ; web-push Node poussé le 2025-03-17, dernière release v3.6.5 d'août 2023 (maintenance lente, lib de référence mais à surveiller) ; ntfy poussé le 2026-07-04, 31 400 étoiles, mainteneur principal binwiederhier ; Gotify poussé le 2026-07-06, 15 200 étoiles ; listmonk v6.2.0 publiée le 2026-06-26, 22 000 étoiles, mainteneur knadh ; Apprise poussé le 2026-07-07, 16 800 étoiles.
- Pile technique : Web Push : simple bibliothèque côté serveur, disponible dans les 4 langages cibles (Node MPL-2.0, Python MPL-2.0, PHP MIT, Go MIT), plus service worker JS côté client ; aucune base de données dédiée (les abonnements push se stockent dans la BDD de l'application). listmonk : Go, binaire unique + PostgreSQL. ntfy : Go, binaire unique, cache SQLite. Gotify : Go + WebSocket. Le choix de la lib web-push suit la stack de l'application hôte ; PHP et Python sont les mieux maintenues en 2026.
- Hébergement : Entièrement auto-hébergeable en France/UE via docker compose : listmonk fournit un docker compose officiel (app + PostgreSQL) ; ntfy et Gotify ont des images Docker officielles. Web Push natif ne requiert AUCUN service supplémentaire : le backend de la PWA signe les envois avec une paire de clés VAPID auto-générée et poste en HTTPS vers les endpoints des navigateurs. Nuance décisive et incontournable : le transport passe par les serveurs de push des éditeurs de navigateurs (Google FCM pour Chrome, Mozilla autopush pour Firefox, Apple pour Safari, Microsoft WNS pour Edge). C'est structurel au standard, sans inscription ni SDK tiers, et le contenu est chiffré de bout en bout (RFC 8291) : les opérateurs ne voient que des métadonnées de trafic. Point dur restant : la délivrabilité du courriel listmonk exige un relais SMTP correct (SPF/DKIM), à prévoir chez un hébergeur FR/UE.
- Couverture du besoin : agenda : Non couvert par ces briques : les flux iCal et RSS doivent être produits par l'application ou le CMS communal ; les briques évaluées ne font que diffuser et notifier.
  signalement : Hors périmètre (Open311 relève de la brique applicative) ; ntfy ou Gotify peuvent en revanche notifier les agents municipaux en interne à chaque nouveau signalement (simple POST HTTP).
  notifications : Couvert : Web Push VAPID natif pour les citoyens (tous navigateurs majeurs ; iOS >= 16.4 uniquement si la PWA est ajoutée à l'écran d'accueil ; Declarative Web Push depuis iOS 18.4 sans service worker, rétrocompatible) ; listmonk pour le courriel en masse ; RSS sortant + publication Mastodon via API pour la rediffusion.
  back-office : Partiel : listmonk offre un back-office de campagnes courriel abouti (i18n fr) ; la saisie unique diffusée multicanal (push + courriel + RSS + Mastodon) demande une petite couche de glue dans l'application (webhook ou cron appelant l'API listmonk, la lib web-push et POST /api/v1/statuses de Mastodon).
- Standards (iCal, RSS, Open311, Web Push) : Web Push est un vrai standard IETF/W3C : RFC 8030 (protocole), RFC 8291 (chiffrement du contenu), RFC 8292 (VAPID). Support réel 2026 : Chrome, Edge, Firefox (desktop + Android), Safari macOS depuis 16.1, iOS/iPadOS depuis 16.4 (mars 2023) exclusivement pour les web apps ajoutées à l'écran d'accueil, mêmes clés VAPID partout, sans compte Apple Developer (source WebKit). Declarative Web Push (JSON sans service worker) livré dans iOS/iPadOS 18.4 (mars 2025) et macOS 15.5/Safari 18.5, devenu W3C Working Draft, rétrocompatible avec les serveurs Web Push classiques. Apple a renoncé en mars 2024 à supprimer les web apps d'écran d'accueil dans l'UE : les PWA et leur push fonctionnent en France. Avec iOS 26, tout site ajouté à l'écran d'accueil s'ouvre par défaut comme web app, ce qui réduit la friction. RSS et iCal sortants : triviaux à générer côté application. Mastodon : API REST documentée (POST /api/v1/statuses). ntfy implémente lui-même Web Push (RFC 8030) et sert de serveur UnifiedPush.
- Français : Les libs web-push sont du code sans UI (non concernées). listmonk : interface d'administration traduite en français (fichiers i18n/fr.json et fr-CA.json vérifiés dans le dépôt). ntfy et Gotify : documentation en anglais uniquement ; état de la traduction française de leurs UI non vérifié. Je n'ai trouvé aucune alternative libre française établie à PanneauPocket/IntraMuros : le marché français des apps municipales est dominé par des solutions propriétaires (constat des comparatifs 2026), ce qui conforte l'intérêt du projet mais signifie zéro retour d'expérience documenté de mairie française sur ce montage précis.
- Ergonomie : Pour les citoyens, l'ergonomie est celle de la PWA communale à construire : les briques n'imposent rien. Friction principale : sur iPhone, l'habitant doit ajouter la PWA à l'écran d'accueil (Partager > Sur l'écran d'accueil) avant de pouvoir activer les notifications ; un tutoriel illustré est indispensable. ntfy et Gotify ont des UI pensées pour des utilisateurs techniques (abonnement à des topics, app à installer hors App Store pour Gotify qui n'a pas d'app iOS officielle) : inadaptées au grand public. UnifiedPush ne concerne que les Android dé-googlisés (~35 apps compatibles début 2026, public de niche). listmonk a un dashboard moderne, adapté à un agent communal pour le canal courriel.
- Risques :
  - Transport push structurellement dépendant des serveurs de push de Google/Apple/Mozilla/Microsoft : contenu chiffré de bout en bout (RFC 8291) mais métadonnées de trafic visibles ; aucune alternative standard n'existe, à assumer et documenter dans le registre RGPD.
  - iOS : push impossible sans installation de la PWA sur l'écran d'accueil (iOS >= 16.4) ; adoption réduite chez les habitants sur iPhone sans accompagnement ; des développeurs rapportent des notifications iOS qui cessent parfois silencieusement (fiabilité à monitorer).
  - La lib Node web-push est en maintenance lente (dernière release août 2023, dernier push mars 2025) : préférer PHP (web-push-php) ou Python (pywebpush), activement maintenus en 2026.
  - Montage = assemblage, pas produit fini : la couche de glue (saisie unique -> push + listmonk + RSS + Mastodon) et son maintien dans le temps reposent sur le prestataire ; risque classique de PWA orpheline.
  - ntfy auto-hébergé n'est PAS une solution pour notifier les citoyens sur iOS : son app iOS exige un relais via ntfy.sh et Apple APNs (upstream-base-url, doc officielle), et l'abonnement à des topics est trop technique pour le grand public.
  - Délivrabilité courriel listmonk : nécessite un relais SMTP bien configuré (SPF/DKIM/DMARC) ; l'auto-hébergement complet d'un MTA est le point le plus fragile du montage.

## Méthode et limites

Recherches menées en français et en anglais (les sources anglophones renvoient quasi exclusivement vers les sources françaises, ce marché étant franco-français). ## 1. PanneauPocket (CityWall SAS)

Le leader rural : environ 14 000 à 15 000 collectivités équipées, 8 millions d'habitants couverts. Métaphore fondatrice : le panneau lumineux municipal transposé dans la poche.

* Fonctionnalités : panneaux d'information et d'alerte (texte + images + PDF + liens), agenda avec géolocalisation (visibilité 30 km), notifications push temps réel, signalement citoyen géolocalisé avec photo (module récent), boîte à idées et sondages via l'extension modulaire « PanneauPocket+ », annuaire (associations, commerces, santé, numéros d'urgence), comptes de sous-entités (écoles, bibliothèques, associations), comptes privés (élus, agents), widget et flux RSS pour le site communal, partage Facebook/X en un clic, compatibilité panneaux lumineux et écrans, statistiques d'ouverture.
* Modèle économique : abonnement annuel par strate de population. Gratuit à vie pour les communes de moins de 100 habitants, 130 €/an pour les communes AMRF de moins de 1000 habitants, environ 350 €/an constaté pour une commune d'environ 3000 habitants (Puttelange-aux-Lacs), sur devis au-delà de 3000 habitants. Pour 7000 habitants : non publié, ordre de grandeur plausible de quelques centaines à environ un millier d'euros par an.
* Diffusion : App Store (4,6/5, 118 000 avis, v6.2.0 de mai 2026), Google Play (4,5/5, plus d'un million de téléchargements), version web app.panneaupocket.com pour les non-équipés.
* Ergonomie : la plus radicalement simple du marché. Pas de compte, pas de donnée personnelle collectée (« le développeur ne collecte aucune donnée »), choix de la commune par favoris (cœur), fil unique de panneaux qui défilent en carrousel, ajout d'événements au calendrier du téléphone, intégration Plans/Google Maps/Waze. Faiblesses relevées par Filomea : pas de recherche, tri limité, messages courts, aucune déclaration RGAA publique.

## 2. IntraMuros (IntraMuros SAS)

L'acteur participatif : plus de 8000 communes et 200 EPCI, environ 1 million de téléchargements. Positionnement territoire (commune + intercommunalité).

* Fonctionnalités : fil d'actualités, alertes notifiées, agenda des événements du territoire, signalement avec photo (jusqu'à 5 photos), géolocalisation et envoi en mairie, sondages, annuaire (associations, commerces, santé, services), menus de cantine, horaires des services (médiathèques), actes administratifs, journal communal, abonnement aux notifications d'une association ou d'un commerce.

Référence publique 2020 : environ 60 €/mois HT pour 5000 habitants, soit 720 €/an. Inclus : contenu illimité, assistance, formations illimitées, maintenance, widget. Seule option payante : la clé API.
* Diffusion : App Store (4,6/5, environ 39 000 avis, v7.10.6 du 8 juin 2026, produit très actif), Google Play, widget de duplication vers le site communal. Pas de version web citoyenne complète confirmée.
* Ergonomie : navigation par onglets (actualités, agenda, services, signalement), notifications ciblées par abonnement. Point différenciant majeur : seul acteur du panel à publier une déclaration d'accessibilité RGAA (partiellement conforme, 98,18 % des critères RGAA 4.1, audit CECIAA, mise à jour octobre 2024).

## 3. Illiwap (Groupe Diagram)

La verticale tout-en-un : environ 3000 collectivités, 1,5 million d'utilisateurs. Concept de « stations » à suivre (communes, intercos, associations).

* Fonctionnalités (tableau officiel 2025) : actualités avec gestion complète (planification, récurrence, fichiers, rapports d'envoi, brouillons), agenda multi-stations et par proximité, annuaire, points d'intérêt (plus de 200 000 référencés), page internet publique dupliquant l'application pour les habitants sans smartphone, import de flux RSS du site communal, widget d'export (RSS ou iframe), partage Facebook automatique, alertes : notifications instantanées, messages géolocalisés touchant même les non-abonnés présents dans une zone avec rayon d'émission paramétrable, alertes SMS avec listes de diffusion (personnes vulnérables, parents d'élèves), protocole d'alerte intégrable au Plan Communal de Sauvegarde, participation : sondages complets (questions illimitées, résultats graphiques, export), signalement citoyen, boîte à idées, et surtout un vrai back-office de gestion des signalements : suivi de l'évolution du traitement, réponse à l'usager par commentaire visible dans son application, édition d'un bon PDF transférable au service concerné, routage par catégorie vers une adresse électronique dédiée. Stations liées illimitées avec administrateurs délégués, station privée interne, statistiques, API et interconnexions en option.
* Modèle économique : abonnement annuel fixé selon le nombre d'habitants, non publié, sur devis au-delà de 5000 habitants (donc sur devis pour 7000 habitants). Inclus : fonctions de base, maintenance, assistance, hébergement. Hébergement France chez Scaleway (DC3, Vitry-sur-Seine).
* Diffusion : App Store (4,6/5, 36 000 avis, v4.11.0 publiée début juillet 2026), Google Play (4,7/5, plus de 25 000 avis, plus de 500 000 téléchargements), page web publique par station.

## 4. CityAll (Lumiplan)

L'application mutualisée d'un industriel de l'affichage (50 ans d'existence, panneaux lumineux) : plus de 1000 communes adhérentes.

* Fonctionnalités : actualités de la mairie et relais des autres médias (posts Facebook, horaires), alertes temps réel notifiées, agenda des événements, signalements citoyens de dysfonctionnements sur la voie publique avec gestion back-office (taux de traitement), sondages pour associer les habitants aux décisions, annuaire et numéros d'urgence, météo, démarches en ligne, mise en avant des associations, abonnement à plusieurs villes, synergie native avec les panneaux lumineux Lumiplan (saisie unique app + panneau).
* Modèle économique : non publié, annoncé « jusqu'à 5 fois inférieur à une application sur mesure » pour les communes de moins de 10 000 habitants. Point de repère : l'application dédiée CityOne du même éditeur démarre à 1500 €/an, CityAll (mutualisée) se situe donc en dessous.
* Diffusion : App Store (3,8/5, seulement 110 avis, v7.3 d'avril 2026) et Google Play. Pas de version web citoyenne identifiée.
* Ergonomie : page d'accueil et design refondus depuis 2021, mode carrousel de l'information, interface décrite comme très simple. C'est toutefois la note store la plus basse du panel (des utilisateurs signalent des problèmes de navigation), et aucune déclaration RGAA trouvée.

## 5. Mon Village (startup bretonne soutenue par Ouest-France)

* Modèle économique : abonnement annuel ajusté à la taille de la commune, montants non publiés, gratuit pour les habitants et pour les acteurs locaux (associations, commerces), livré clé en main (formation, communication, assistance). Hébergement des données en France, conformité RGPD affichée.
* Diffusion : App Store (4,7/5 mais seulement 1900 avis, v1.52.2 d'avril 2026), Google Play, application web app.monvillage.bzh.
* Ergonomie : fil d'actualité type réseau social avec logique d'abonnement par acteur local, favoris, espace personnel « Mon espace », navigation redessinée. ## Conclusion 1 : le socle fonctionnel attendu par une mairie (les 5 fonctions que toutes offrent)

1. Actualités : un fil unique d'informations publié par la mairie, avec notifications push en temps réel, texte + image + PDF + lien.
2. Alertes : messages urgents à forte visibilité (météo, coupures, sécurité, PCS), distincts des actualités courantes, notifiés immédiatement.
3. Agenda des événements locaux, consultable par date et catégorie, avec ajout au calendrier du téléphone chez les meilleurs.
4. Signalement citoyen : photo + géolocalisation + catégorie envoyées en mairie, avec, chez les plus aboutis (Illiwap, CityAll, IntraMuros), un back-office de suivi : statuts de traitement, routage par catégorie vers le bon service ou la bonne adresse mail, réponse visible par l'habitant. C'est exactement le périmètre Open311 visé par le projet.
5. Annuaire local : services municipaux, associations, commerces, professionnels de santé, numéros d'urgence, horaires.

Une sixième fonction est quasi universelle : sondages et boîte à idées (native chez IntraMuros, Illiwap, CityAll, Mon Village, en module optionnel chez PanneauPocket). S'y ajoute partout la promesse de saisie unique multicanal : une publication alimente l'app, le widget ou RSS du site communal, les réseaux sociaux, et souvent les panneaux lumineux.

## Conclusion 2 : ce qui fait leur qualité perçue (le niveau à égaler, sans copier)

* Zéro friction pour l'habitant : aucune création de compte, anonymat, gratuité, absence de publicité, choix de la commune en deux gestes avec favoris multi-communes. C'est l'argument numéro un de PanneauPocket et d'Illiwap, et une exigence directe pour une alternative libre (PWA sans inscription).
* Le push comme colonne vertébrale : la valeur perçue par les élus est « l'information qui va vers l'habitant », notifications fiables, immédiates, mais maîtrisées par abonnement aux canaux choisis (stations, sous-entités, catégories).
* Simplicité radicale de l'interface : un fil unique de cartes ou de panneaux qui se parcourt verticalement ou en carrousel, patterns familiers (cartes illustrées, badges de contenus non lus, filtres de catégories, pastilles de statut sur les signalements), lisible par des publics seniors, français simple.
* Back-office « mairie sans informaticien » : SaaS pur, rien à installer, publication en quelques minutes depuis un navigateur, formation et assistance incluses, multi-administrateurs avec droits, statistiques d'audience simples. La maintenance nulle côté commune est un argument de vente central : une alternative libre doit l'égaler via un hébergement infogéré docker compose chez un tiers.
* Délégation multi-émetteurs : sous-comptes illimités pour écoles, associations, commerces (stations liées Illiwap, sous-entités PanneauPocket, canaux Mon Village), qui démultiplient le contenu sans charge pour la mairie.
* Confiance affichée : hébergement en France (OVHcloud, Scaleway), RGPD mis en avant, minimisation des données (Illiwap efface les données de signalement au bout d'un mois).
* Version web de repli pour les habitants sans smartphone (app.panneaupocket.com, page internet Illiwap, app.monvillage.bzh) : un point où une PWA libre est structurellement supérieure.

Failles exploitables par une alternative libre : accessibilité RGAA quasi absente (seul IntraMuros publie une déclaration, 98,18 % RGAA 4.1), opacité tarifaire au-delà de 3000 à 5000 habitants (budget annuel réel pour 7000 habitants : de l'ordre de 700 à 1500 €/an et davantage avec options SMS ou API), verrouillage propriétaire sans export standard (aucun ne parle iCal, RSS entrant excepté, ni Open311), et absence de PWA installable (tous misent sur les stores).

Points clés :
- Socle fonctionnel commun aux 5 applications (la parité minimale attendue par une mairie) : 1) actualités avec notifications push, 2) alertes urgentes, 3) agenda des événements, 4) signalement citoyen photo + géolocalisation + catégorie avec suivi, 5) annuaire local ; les sondages/boîte à idées et la saisie unique multicanal (app, widget site, réseaux sociaux, panneaux lumineux) complètent ce socle chez presque tous.
- Qualité perçue = zéro friction : pas de compte, anonymat, gratuité et absence de publicité pour l'habitant, choix de la commune en deux gestes, favoris multi-communes ; le push fiable et maîtrisé par abonnement à des canaux est la colonne vertébrale de la proposition de valeur (« l'information va vers l'habitant »).
- Patterns d'interface à égaler : fil unique de cartes/panneaux (défilement vertical ou carrousel), badges de non-lus, filtres par catégorie, pastilles de statut sur les signalements (à traiter, en cours, résolu), sous-comptes délégués aux écoles/associations/commerces (stations liées Illiwap, sous-entités PanneauPocket), ajout d'événements au calendrier du téléphone.
- Modèle économique : abonnement annuel SaaS par strate de population, systématiquement sur devis pour 7000 habitants ; repères publics : PanneauPocket gratuit <100 hab, 130 €/an AMRF <1000 hab, ~350 €/an constaté à ~3000 hab ; IntraMuros ~720 €/an HT pour 5000 hab (référence 2020, grille 5001-7500 hab non publiée) ; CityOne (Lumiplan dédié) dès 1500 €/an, CityAll mutualisé en dessous ; Illiwap et Mon Village non publiés. Budget réaliste 7000 hab : ~700 à 1500 €/an hors options SMS/API.
- Poids de marché et activité vérifiée (pas de dépôts publics, vérification via app stores) : PanneauPocket ~14-15 000 collectivités (iOS 4,6/5, 118 000 avis, v6.2.0 mai 2026), IntraMuros 8000 communes (4,6/5, 39 000 avis, v7.10.6 juin 2026), Illiwap 3000 collectivités (4,6/5, v4.11.0 juillet 2026), CityAll >1000 communes (3,8/5, 110 avis, v7.3 avril 2026), Mon Village >50 collectivités (4,7/5, 1900 avis, v1.52.2 avril 2026) : les cinq sont activement maintenues.
- Le back-office « mairie sans informaticien » est décisif : SaaS pur, zéro installation, publication en minutes, formation et assistance incluses, multi-administrateurs avec droits, statistiques simples ; Illiwap a le back-office signalements le plus abouti (statuts de traitement, réponse visible par l'habitant, bon PDF transférable, routage par catégorie vers une adresse mail de service), ce qui correspond au périmètre Open311 du projet.
- Failles exploitables par une alternative libre : accessibilité RGAA quasi absente (seul IntraMuros publie une déclaration, 98,18 % RGAA 4.1), opacité tarifaire, aucun standard ouvert (pas d'iCal, pas d'Open311, RSS seulement en widget sortant ou import), pas de PWA installable (dépendance totale aux stores, la version web n'est qu'un repli), verrouillage des données ; tous s'appuient en revanche sur l'hébergement France (OVHcloud, Scaleway) et le RGPD comme argument de confiance, niveau à égaler.
- Précision d'attribution : « Mon Village » n'est pas éditée par Ouest-France mais par une startup bretonne fondée par Gurwan Bertho ; Ouest-France est entré au capital (montant non dévoilé, avril 2025) et apporte Infolocale/Panorapresse ; sa diffusion (~50 collectivités) reste marginale face aux quatre autres.
- Informations introuvables malgré recherche : grilles tarifaires exactes IntraMuros 5001-7500 hab (PDF sur demande uniquement), tarifs Illiwap (sur devis dès 5000 hab), tarifs CityAll et Mon Village (jamais publiés), tarif PanneauPocket au-delà de 3000 hab ; toute estimation budgétaire pour 7000 habitants doit donc être confirmée par demande de devis.

## Standards et obligations

1) OPEN311 GEOREPORT V2. La spécification de référence (wiki.open311.org/GeoReport_v2/) définit une API REST avec 6 endpoints : GET /services.[format] (liste des catégories), GET /services/[service_code].[format] (définition des attributs d'un service), POST /requests.[format] (création d'un signalement), GET /tokens/[token_id].[format] (résolution d'un token en service_request_id), GET /requests.[format] (interrogation) et GET /requests/[service_request_id].[format]. Champs obligatoires au POST : service_code, une localisation (lat et long, OU address_string, OU address_id), jurisdiction_id si multi-juridiction, et les attribute[code] exigés par la service definition (datatypes : string, number, datetime, text, singlevaluelist, multivaluelist). Champs optionnels normalisés : description (max 4000 caractères), media_url (photo), email, first_name, last_name, phone, device_id. XML est le format obligatoire, JSON optionnel, UTF-8 partout, dates ISO 8601 avec fuseau. Gestion des statuts : le standard ne connaît que DEUX statuts, open et closed, ce qui est sa limite la plus structurante pour un back-office à trois états (à traiter, en cours, résolu) : il faudra soit mapper (à traiter et en cours vers open), soit adopter les extensions de FixMyStreet qui ajoutent GET/POST service_request_updates (mises à jour horodatées avec statuts étendus, description et média) documentées sur le wiki mySociety. Autres limites connues : requêtes limitées par défaut à 90 jours ou 1000 résultats, pas de mécanisme d'authentification citoyen standardisé (simple api_key), pas de notification push ni d'abonnement, pas d'upload direct de photo (media_url est une URL). État d'adoption en 2026 : le standard est FIGÉ mais pas mort. Le dépôt du wiki officiel (open311/open311.github.io) n'a plus reçu de push depuis le 24 mai 2022 (50 issues ouvertes), la spec n'évolue plus. En revanche il reste le standard d'interopérabilité de fait du signalement citoyen : FixMyStreet (mySociety) le supporte en client et partiellement en serveur, dépôt très actif (dernier push le 7 juillet 2026, release v6.0 du 15 novembre 2024), et Mark-a-Spot (Drupal 11 + PWA Vue.js, conforme Open311) est également actif (dernier push le 26 juin 2026). Adoption historique documentée : villes US, Toronto, Allemagne (Mark-a-Spot), UK (FixMyStreet), Helsinki et le programme européen CitySDK. Conclusion pratique : implémenter le cœur GeoReport v2 pour l'interopérabilité, plus les extensions service_request_updates à la FixMyStreet pour les statuts riches.

2) ICAL (RFC 5545) ET RSS POUR L'AGENDA. iCalendar (media type text/calendar, extension .ics) est défini par la RFC 5545. Bonnes pratiques normatives (section 5, pratiques recommandées) : plier les lignes de plus de 75 octets, ne jamais tronquer un SUMMARY au milieu d'une séquence UTF-8, consolider les doublons RRULE/RDATE. Pour un calendrier PUBLIÉ par abonnement, la RFC 7986 (New Properties for iCalendar, 2016) apporte les propriétés clés : NAME et DESCRIPTION du calendrier, REFRESH-INTERVAL (intervalle minimal de polling suggéré, la RFC note qu'une mise à jour quotidienne suffit dans la plupart des cas pour un calendrier public), SOURCE (URL canonique du flux) et COLOR. Bonnes pratiques opérationnelles : UID stable et unique par événement (indispensable pour que les mises à jour remplacent l'événement au lieu de le dupliquer), METHOD:PUBLISH, VTIMEZONE avec TZID Europe/Paris plutôt que des heures flottantes, servir en HTTPS avec Content-Type text/calendar, et valider le flux avec le validateur icalendar.org. Précédent municipal français documenté : la ville de Saran publie un flux iCal glissant (fenêtre de -1 mois à +2 mois) testé avec Google Agenda, Outlook et iPhone (fiche Villes Internet). Pour RSS : suivre la RSS 2.0 Specification et surtout le RSS Best Practices Profile du RSS Advisory Board (guid avec permalien stable par événement, pubDate au format RFC 822, une seule enclosure par item, lien vers la page de l'événement), et valider avec le W3C Feed Validator. Atom (RFC 4287) est une alternative plus rigoureuse mais RSS 2.0 reste le plus interopérable.

3) WEB PUSH (RFC 8030) ET VAPID (RFC 8292). Architecture à trois acteurs : le navigateur (user agent) s'abonne auprès d'un push service (opéré par l'éditeur du navigateur : Mozilla autopush, endpoints FCM pour Chrome, Apple pour Safari), et le serveur applicatif de la commune envoie ses messages en POST HTTP sur l'URL d'endpoint fournie par l'abonnement. RFC 8030 : l'en-tête TTL (durée de rétention en secondes) est OBLIGATOIRE, le push service répond 400 s'il est absent, en-têtes optionnels Urgency et Topic (remplacement de message). RFC 8292 (VAPID) : le serveur s'auto-identifie via un JWT signé ES256 (courbe P-256) transmis dans Authorization: vapid t=...,k=..., avec une claim sub de contact (mailto:). VAPID permet la restriction d'abonnement : le push service peut rejeter les messages non signés par la clé attendue sans contacter l'appareil, ce qui protège contre la fuite d'endpoints. Aucun compte auprès des push services n'est nécessaire grâce à VAPID. RFC 8291 complète : chiffrement de bout en bout obligatoire du payload en aes128gcm (ECDH P-256 + HKDF SHA-256), payload utile limité à environ 3993 octets en clair (4096 chiffrés). Prérequis serveur concrets : site en HTTPS (les service workers exigent un contexte sécurisé), génération et conservation d'une paire de clés VAPID P-256, stockage des abonnements (endpoint + clés p256dh et auth), purge des abonnements sur réponses 404/410, et une bibliothèque type web-push. Contrainte iOS majeure pour un projet PWA mobile d'abord : depuis iOS/iPadOS 16.4, Web Push fonctionne UNIQUEMENT pour les web apps ajoutées à l'écran d'accueil, et la demande de permission doit répondre à une interaction utilisateur directe (blog WebKit officiel).

4) RGAA. Version en vigueur en 2026 : RGAA 4.1.2, la version 5 est en cours de rédaction avec publication prévue fin 2026 (site officiel accessibilite.numerique.gouv.fr). Une commune de 7000 habitants est pleinement assujettie : l'article 47 de la loi n° 2005-102 du 11 février 2005 vise toutes les personnes morales de droit public sans seuil de taille (sites web, applications mobiles, intranets). Obligations : conformité au RGAA (106 critères, 13 thématiques), publication d'une déclaration d'accessibilité par service, schéma pluriannuel de mise en accessibilité d'une durée maximale de 3 ans décliné en plans d'actions annuels, mention du statut de conformité sur la page d'accueil (totalement, partiellement ou non conforme) et moyen pour l'usager de signaler un défaut d'accessibilité. Sanctions : l'ordonnance n° 2023-859 du 6 septembre 2023 (créant l'article 47-1) et le décret n° 2023-931 du 9 octobre 2023 confient le contrôle à l'Arcom, qui peut prononcer, après mise en demeure, jusqu'à 50 000 euros par service en ligne non accessible, renouvelables tous les 6 mois si le manquement persiste, et jusqu'à 25 000 euros pour manquement aux obligations déclaratives (déclaration, schéma, plan annuel). L'Arcom a annoncé une montée en puissance des contrôles en 2026. Conséquence projet : la conformité RGAA 4.1.2 et les trois documents déclaratifs doivent être prévus dès la spécification, ce qui pèse sur le choix de la base logicielle (thèmes accessibles, HTML sémantique).

5) RGPD APPLIQUÉ AU SIGNALEMENT CITOYEN. Base légale : pour le service de signalement, la base adaptée est la mission d'intérêt public (article 6.1.e RGPD), conformément aux principes rappelés par la CNIL pour les collectivités (guide de sensibilisation RGPD collectivités territoriales) ; le consentement (6.1.a) est la base des notifications Web Push, qui sont par construction opt-in via le navigateur. Données collectées et minimisation : géolocalisation ponctuelle du problème (pas de traçage continu), photo (risque de personnes identifiables ou de plaques : prévoir modération avant publication, recommander le cadrage sur le problème, ne publier publiquement que le strict nécessaire), coordonnées du déclarant (email pour le suivi, à ne jamais publier). Durées de conservation : il n'existe PAS de référentiel CNIL fixant une durée spécifique pour le signalement citoyen (je le dis honnêtement, les référentiels CNIL publiés couvrent RH, social et médico-social, activités commerciales) ; s'appliquent donc le principe général du guide CNIL des durées de conservation (base active limitée à l'objectif, puis destruction, anonymisation ou archivage selon le régime des archives publiques) et le précédent notable de DansMaRue (Ville de Paris) : suppression des données personnelles après 1 an sans utilisation par l'usager. Une politique raisonnable : anonymiser le signalement (conserver la donnée technique du problème, supprimer coordonnées et éléments identifiants des photos) quelques mois après résolution. Autres obligations : DPO obligatoire pour toute commune quelle que soit sa taille (article 37 RGPD, autorité publique), mutualisable via EPCI, centre de gestion ou opérateur public de services numériques (la CNIL a déjà mis en demeure 22 communes sur ce point), registre des traitements, information des personnes (article 13), droits d'accès, rectification et opposition, sécurité (chiffrement, journalisation).

6) HÉBERGEMENT. SecNumCloud n'est PAS obligatoire pour une commune de 7000 habitants : la doctrine cloud au centre s'applique à l'administration centrale et à ses organismes sous tutelle, la FAQ officielle de numerique.gouv.fr indique explicitement que les collectivités ne rentrent pas dans le champ d'application de la circulaire et ne sont pas tenues de recourir au cloud qualifié, tout en y étant invitées volontairement pour les données sensibles. La loi SREN du 21 mai 2024 impose SecNumCloud pour les données d'une sensibilité particulière de l'État, et son décret d'application a assujetti six GIP (Acteurs Publics, Banque des Territoires). Un amendement adopté en commission des lois le 25 février 2026 (proposition Latombe) étendrait l'obligation aux collectivités et intercommunalités de PLUS de 30 000 habitants pour leurs données sensibles ; il n'était pas encore voté en séance (examen prévu le 26 mars 2026) et ne concernerait de toute façon pas une commune de 7000 habitants. Un agenda public et des signalements de voirie ne relèvent pas des données d'une sensibilité particulière au sens SREN (gestion de crise, ordre public, santé). Recommandations effectives : hébergement en France ou UE chez un prestataire non soumis à des transferts hors UE problématiques (chapitre V RGPD), contrat de sous-traitance article 28 RGPD, mesures de sécurité CNIL (le guide CNIL et cybermalveillance.gouv.fr sur les obligations des collectivités sert de référence), sauvegardes, HTTPS. SecNumCloud reste un plus volontaire, pas une exigence. Aucune obligation HDS (pas de données de santé).

Points clés :
- Open311 GeoReport v2 est un standard figé (dernier push du wiki officiel : mai 2022) mais toujours utilisé comme couche d'interopérabilité de fait en 2026 par des projets actifs (FixMyStreet, push du 7 juillet 2026 ; Mark-a-Spot, push du 26 juin 2026) ; son cœur impose XML, service_code + localisation au POST, et ne connaît que deux statuts open/closed : les trois statuts du back-office (à traiter, en cours, résolu) exigent un mapping ou les extensions service_request_updates popularisées par FixMyStreet.
- Pour l'agenda : iCal RFC 5545 avec UID stables, VTIMEZONE Europe/Paris et lignes pliées à 75 octets, complété par la RFC 7986 (NAME, SOURCE, REFRESH-INTERVAL, un rafraîchissement quotidien suffit pour un calendrier public) ; côté RSS suivre le RSS Best Practices Profile du RSS Advisory Board (guid permalien stable, pubDate RFC 822, une enclosure par item) et valider les deux flux (validateur icalendar.org, W3C Feed Validator).
- Web Push : en-tête TTL obligatoire (RFC 8030), identification serveur par JWT ES256/P-256 VAPID sans compte auprès des push services (RFC 8292), chiffrement obligatoire du payload en aes128gcm limité à ~4 Ko (RFC 8291) ; prérequis serveur : HTTPS, paire de clés VAPID, stockage des abonnements (endpoint, p256dh, auth), purge sur 404/410 ; contrainte forte pour une PWA : sur iOS (16.4+) le push ne fonctionne que si la web app est ajoutée à l'écran d'accueil et après interaction utilisateur explicite.
- RGAA : version 4.1.2 en vigueur (v5 attendue fin 2026), applicable à toute commune sans seuil de taille (art. 47 loi 2005-102) ; obligations : conformité aux 106 critères, déclaration d'accessibilité, schéma pluriannuel de 3 ans maximum avec plans annuels, mention de conformité en page d'accueil ; sanctions prononcées par l'Arcom depuis l'ordonnance 2023-859 et le décret 2023-931 : jusqu'à 50 000 € par service non conforme renouvelables tous les 6 mois, et 25 000 € pour défaut des obligations déclaratives.
- RGPD : base légale mission d'intérêt public (art. 6.1.e) pour le signalement, consentement pour les notifications push ; DPO obligatoire même pour une commune de 7000 habitants (art. 37, mutualisable via EPCI ou centre de gestion) ; photos et géolocalisation à minimiser avec modération avant publication ; aucune durée de conservation officielle CNIL spécifique au signalement citoyen, appliquer le principe de proportionnalité du guide CNIL des durées de conservation, avec le précédent DansMaRue (Paris) : suppression après 1 an sans utilisation.
- Hébergement : SecNumCloud n'est pas obligatoire pour une commune de 7000 habitants (doctrine cloud au centre limitée à l'État selon la FAQ numerique.gouv.fr ; loi SREN et son décret visant l'État et six GIP ; l'amendement de février 2026 en cours d'examen ne viserait que les collectivités de plus de 30 000 habitants) ; les exigences effectives sont un hébergeur France/UE sans transfert hors UE problématique, un contrat art. 28 RGPD et les mesures de sécurité CNIL, SecNumCloud restant une option volontaire.
- Points d'incertitude signalés honnêtement : pas de statistiques consolidées d'adoption Open311 en 2026 (uniquement des preuves d'activité des implémentations), pas de référentiel CNIL dédié au signalement citoyen, et statut législatif non définitif de l'extension SecNumCloud aux collectivités (adopté en commission le 25 février 2026, séance prévue le 26 mars 2026, issue non vérifiable dans les sources consultées).
